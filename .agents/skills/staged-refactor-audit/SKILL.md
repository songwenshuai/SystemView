# 暂存区重构拆分审查

使用此技能审查当前 Git 暂存区中的重构和拆分变更。审查目标是确认严格等价部分没有丢失既有行为，没有遗漏迁移内容，也没有把原有错误带入新结构；对用户或文档已说明的正确性、可预测性、可移植性改进，确认其是否属于根因层级的合理语义收敛，而不是仅因行为不完全等价就判为问题。

## 审查边界

- 只审查已暂存内容，以 index 为唯一审查对象。
- 不运行 `git add`，不修改暂存区，不恢复、不 stash、不 reset。
- 不把未暂存工作区内容当作审查依据；如发现工作区和暂存区同时修改同一文件，必须明确区分。
- 不运行编译或测试命令；需要验证时，请用户执行精确命令并提供输出。
- 除非用户明确要求修复，否则采用代码审查姿态输出问题清单，不直接编辑文件。

## 等价性判定标准

- 默认按行为等价审查；如果用户明确说明“不要求完全等价”或暂存文档、计划、PR 描述中声明本批包含正确性改进、语义收敛、错误处理收敛、可移植性修复，则按“重构 + 合理改进”审查。
- 合理语义改进必须满足：
  - 修复位于根因层级，而不是调用点条件补丁。
  - 消除未定义行为、未对齐访问、端序依赖、静默错误、重复真相源、错误计数缺失、状态不一致或其他可验证缺陷。
  - 新语义更明确、更可预测，并且不隐藏失败。
  - 影响 public API、CLI、文件格式、协议、序列化或错误输出时，必须有文档或用户确认。
  - 构建清单、调用方和验证项必须同步。
- 行为差异分类：
  - `确定问题`：意外改变外部行为、丢失功能、破坏公开契约、遗漏迁移、引入条件补丁或未声明的兼容性破坏。
  - `已确认的合理语义改进`：用户或暂存文档已明确接受，且实现从根因层级提升正确性、可预测性或可移植性。
  - `需要用户确认的设计意图`：改进方向可能合理，但当前暂存内容或用户上下文没有明确授权。
  - `需要编译测试验证的风险`：逻辑上成立但仍依赖构建、测试、golden 输出或平台验证。
- 如果用户已在本轮确认某个差异不要求完全等价，后续报告不得继续把同一差异列为“需要用户确认”，应改列为“已确认的合理语义改进”或验证风险。

## 输入识别

1. 收集暂存区概况：
   - `git status --short`
   - `git diff --cached --name-status --find-renames`
   - `git diff --cached --stat`
   - `git diff --cached --check`

2. 如果没有暂存变更，停止审查并告知用户没有已暂存内容。

3. 按状态建立文件清单：
   - `A`：新增文件，必须读取新增文件的完整暂存内容。
   - `M`：修改文件，必须读取该文件完整暂存 diff。
   - `D`：删除文件，必须读取删除前的完整内容。
   - `R`：重命名或移动，必须读取旧路径内容、新路径暂存内容和完整 diff。
   - `C`：复制，必须读取来源内容、目标暂存内容和完整 diff。

## 强制读取规则

- 对每个暂存文件逐一分析，不允许只抽样查看。
- 修改文件使用 `git diff --cached --no-ext-diff -- <path>` 读取完整 diff。
- 新增文件使用 `git show :<path>` 读取完整暂存内容。
- 删除文件使用 `git show HEAD:<path>` 读取删除前完整内容。
- 重命名或移动文件同时读取：
  - `git show HEAD:<old-path>`
  - `git show :<new-path>`
  - `git diff --cached --no-ext-diff -- <old-path> <new-path>`
- 如果 diff 很大，分段读取仍必须覆盖完整内容，并记录已覆盖的范围。
- 对拆分变更，必须把删除内容、旧文件移出内容、新文件新增内容、构建配置变更放在同一组里交叉比对。

## 审查重点

### 行为等价

- 比较重构前后的控制流、数据流、错误路径和资源生命周期。
- 检查所有原有分支、边界条件、默认值、配置项、日志、断言和错误返回是否仍存在。
- 检查 public API、CLI 参数、配置键、序列化格式、文件格式、协议字段和外部可见行为是否被意外改变。
- 检查顺序敏感逻辑：初始化顺序、销毁顺序、锁顺序、注册顺序、回调顺序和遍历顺序。
- 检查异常、错误码、返回值、空值处理、越界处理和清理路径是否保持一致。
- 对声明为合理语义改进的差异，检查其根因、影响范围、契约说明和验证要求，而不是机械要求旧行为完全保留。

### 拆分完整性

- 确认每个被迁移的函数、类型、宏、常量、静态变量和 helper 都有明确归属。
- 检查文件级静态状态迁移后是否仍满足所有权和生命周期约束。
- 检查头文件拆分是否保持接口边界清晰：公开接口放在公开头文件，内部接口放在 internal 头文件或实现文件。
- 检查 include 依赖是否完整且没有引入循环依赖。
- 检查构建系统是否纳入新增源文件，是否移除删除文件，是否同步平台条件和安装导出配置。
- 检查测试、示例、脚本和文档引用的路径、符号和模块名是否同步更新。

### 功能丢失

- 对删除文件和移出代码逐项确认替代位置或删除理由。
- 对旧实现中的所有用户可见路径建立映射：旧入口 -> 新入口。
- 标记无法在新结构中找到对应实现的分支、配置、错误处理、清理动作和边界检查。
- 标记因为拆分导致的默认初始化缺失、回调未注册、对象未释放、状态未复制或状态重复维护。

### 错误引入

- 检查新抽象是否改变值语义、所有权语义、线程语义或错误传播语义。
- 检查参数顺序、单位、编码、大小端、路径规范化和类型转换是否发生误用。
- 检查新 helper 是否过度泛化、参数膨胀、布尔标志过多或隐藏调用前置条件。
- 检查并发代码是否出现跨模块锁、锁顺序改变、TOCTOU、悬空引用、UAF 或泄漏。
- 检查拆分后重复逻辑是否产生多个真相源。

### 旧错误延续

- 如果重构触及已知问题、注释中标记的缺陷、历史 TODO、调用方绕行逻辑或测试覆盖的失败场景，必须确认新结构是否根本消除了问题。
- 不接受把旧错误搬到新文件、添加包装层绕过问题或在调用点增加条件补丁。
- 如果此次变更声明只是重构，不应要求额外功能；但对已触及的错误路径，必须指出仍未修复的根因。

## 对比方法

- 优先把移动和拆分视为语义迁移，而不是独立新增与删除。
- 对大型拆分，建立迁移表：
  - 旧符号或旧路径
  - 新符号或新路径
  - 行为是否保持
  - 风险点
- 使用 `rg` 搜索旧符号、新符号、旧路径和新路径，确认调用方、构建配置和文档引用全部同步。
- 对表驱动规则、枚举、关键字、错误码、操作符和平台符号，确认规则集合没有缺项、重复项或顺序错误。
- 对公共接口变化，搜索所有调用方并检查每个调用点是否按新契约调整。

## 输出要求

- 发现问题时，按严重程度列出，格式为：
  - `文件:行` 或 `文件`。
  - 问题说明。
  - 为什么该问题会导致行为丢失、错误引入或旧错误延续。
  - 建议的修复方向。
- 如果没有发现问题，明确说明未发现功能丢失、错误引入或旧错误延续，并列出仍需用户编译或测试验证的范围。
- 明确说明审查覆盖了哪些暂存文件状态：新增、修改、删除、重命名、复制。
- 报告必须区分确定问题、已确认的合理语义改进、需要用户确认的设计意图和需要编译测试验证的风险。

# 代码审查

使用此技能审查所有已变更文件，识别质量问题，并实现所有可执行修复。

## 输入

- 用户可提供一个可选的 commit hash。
- 如果用户没有提供 commit hash，审查当前 Git 变更。
- 如果没有 Git 变更，审查用户明确提到的文件，或当前对话中已经编辑过的文件。

## 工作流程

1. 识别变更集合。
   - 有 commit hash 时：检查 `git show <commit-hash>`。
   - 没有 commit hash 时：用 `git diff` 检查当前变更；如果存在已暂存变更，检查 `git diff HEAD`。
   - 记录已修改、新增、删除、重命名和移动的文件。

2. 阅读完整文件上下文。
   - 对变更较小的文件，阅读当前完整文件，并结合 diff 理解变更上下文。
   - 对大范围重写，阅读修改前和当前的完整文件内容。只允许使用只读 `git show` 命令；禁止把历史内容重定向写入文件。
   - 对删除文件，检查删除前的完整内容，理解被移除的行为。
   - 对新增文件，阅读完整的新文件内容。
   - 当删除和新增看起来像未识别的重命名时，先比较路径和内容，再决定是否把它们当作无关文件处理。

3. 审查复用性。
   - 搜索可替代重复新增代码的现有工具函数、辅助函数、共享模块、相邻模式和框架 API。
   - 标记与现有行为重复的新辅助函数。
   - 标记应该使用现有项目抽象的内联逻辑。

4. 审查质量和正确性。
   - 比较修改前后的行为，检查是否丢失分支、功能、边界处理、配置项或错误路径。
   - 基于代码语义验证逻辑，不依赖注释或提交信息。
   - 确认 bug fix 解决根因，并覆盖所有同类情况。
   - 检查变更后的格式是否保持局部对齐约定。
   - 标记冗余状态、参数膨胀、近似重复代码块、抽象泄漏、已有常量或枚举时仍使用原始字符串、以及不必要的包装元素。

5. 审查效率。
   - 查找冗余计算、重复文件读取、重复 API 调用、N+1 模式、遗漏的独立并行机会、热路径阻塞工作、无操作状态更新、TOCTOU 存在性检查、无界内存增长、缺少清理以及过宽的数据加载。

6. 修复可执行发现。
   - 使用仓库现有模式直接实现修复。
   - 对误报直接跳过。
   - 将 diff 限定在被审查的变更集合和必要修复范围内。

7. 进行逻辑验证。
   - 不在此仓库运行编译或测试命令。
   - 如果需要编译或测试验证，请用户运行精确命令并提供输出。
   - 总结已修复问题、已跳过误报和需要用户手动执行的验证。
